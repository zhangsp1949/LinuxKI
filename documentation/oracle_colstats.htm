LinuxKI Warning

 

Oracle column tracking causing high CPU usage by Oracle processes

Date: 05/13/21

 

Problem

Customer noticed very high CPU usage (>85%) on both nodes of their Oracle RAC DB environment using two new SDFlex 4-socket servers with Platinum 8156 processors.   The DB servers were running RHEL 7.5 and using Oracle 18c.

Investigation

LinuxKI data was collected during the period of high CPU usage.     Analysis on one of the busy Oracle processes shows most of the CPU time was spent in user (Oracle) code.

While HPE does not have source code to review, the name of the found sounds like it’s related to statistics.   

Solution

After consulting with Oracle and showing them the data, Oracle recommended changing _column_tracking_level from the default value of 21 to 1.     After changing the parameter, the CPU usage was reduced from 85-90% to below 13%.

Note, changing hidden parameters in Oracle should be performed with care and after consultation with Oracle.   There may be other cases where the default _column_tracking_level provides a performance benefit.

For more information, please refer to Oracle Document ID 2773012.1, which documents the following problem:

Problem 3: High CPU for SQL statements with too many columns. The short stack shows functions spinning on qosdGetOptDir, qosdInitDirCtx, qosdUpdExprExecStatsRws. There could be high latch waits including GES latches in RAC.

Solution: Set, _column_tracking_level=1 at system level. It’s a dynamic parameter. This is to avoid the extensive column usage tracking which could incur more CPU. The default value was 1 in 11.2 & 12.1 and changed to 21 and higher from 12.2 onwards.
